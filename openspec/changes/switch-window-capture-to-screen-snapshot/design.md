## Context

当前识别设置中的窗口截图实现直接针对目标窗口抓取内容，这在部分由特殊引擎或非标准渲染链路绘制的窗口上不稳定，可能得到空白图、黑屏图或直接失败。由于后续区域标注、模板保存和区域匹配都依赖这张截图，截图入口一旦不稳定，整个识别配置流程都会失去基础。

本次改动不改变截图接口的上层调用方式，而是替换底层实现：先获取当前屏幕快照，再根据目标窗口在屏幕上的位置和尺寸裁切目标图像。约束包括：
- 裁切必须基于窗口矩形而不是窗口客户区逻辑，保持与用户屏幕所见一致。
- 窗口矩形可能部分超出屏幕，因此裁切前必须做边界收敛。
- 识别设置、模板保存和匹配逻辑都继续复用同一个窗口截图入口，避免上层代码大范围改动。
- 这次设计只替换截图来源，不新增新的识别页面行为或模板格式变更。

## Goals / Non-Goals

**Goals:**
- 将窗口截图底层实现切换为整屏快照加窗口矩形裁切。
- 提高对特殊渲染窗口的截图兼容性。
- 保持现有窗口截图调用入口和返回格式尽量不变。
- 明确窗口矩形越界、空区域和截图失败时的处理方式。

**Non-Goals:**
- 不改变识别设置页面的交互流程。
- 不改变模板保存格式或区域匹配结果结构。
- 不在本次设计中扩展多屏独立选择、窗口遮挡恢复或后台无损截图等高级能力。
- 不引入新的图像处理算法，只关注截图来源替换和裁切准确性。

## Decisions

### 1. 使用屏幕快照作为统一截图源，再按窗口矩形裁切目标图像
底层先抓取当前桌面可见区域图像，再依据目标窗口矩形计算裁切框输出窗口截图。

选择原因：
- 对特殊渲染窗口更稳妥，因为绕开了直接读取目标窗口渲染内容的限制。
- 用户看到的屏幕内容与最终截图来源一致，更符合“所见即所得”的识别配置体验。

备选方案：
- 继续优先尝试目标窗口截图，失败后回退到屏幕快照。未采用，因为会保留双路径复杂度，本次目标是明确切换到稳定方案。

### 2. 裁切时使用窗口矩形与屏幕边界求交，避免越界读取
在根据窗口位置和大小裁切前，先将目标矩形限制在实际屏幕快照边界内；如果交集为空，则返回明确错误。

选择原因：
- 部分窗口可能位于屏幕边缘或部分超出可见区域。
- 显式边界收敛可以减少崩溃、越界或无效图像输出。

备选方案：
- 直接假定窗口完全在屏幕内。未采用，因为真实桌面环境下这个假设并不可靠。

### 3. 保持上层窗口截图接口稳定，仅替换底层捕获策略
识别设置页和模板流程继续通过原有的窗口截图接口获取图片，不因为底层实现切换而修改调用语义。

选择原因：
- 改动范围集中在截图底层，前端和模板流程无需跟着调整。
- 这样更容易验证“截图来源变了，但上层行为没变”。

备选方案：
- 新增第二套专用截图接口。未采用，因为会让上层调用分叉，增加维护成本。

## Risks / Trade-offs

- [屏幕快照只能得到当前可见内容，窗口被遮挡时截图会包含遮挡] -> 这与“用户当前所见配置截图”一致，先接受该约束；如需后台无遮挡截图可在后续独立讨论。
- [多屏或高 DPI 环境下坐标映射可能出错] -> 实现时统一以当前屏幕快照坐标系和窗口矩形做换算，并补充边界校验。
- [整屏截图成本高于单窗口截图] -> 当前截图触发频率低，识别设置是交互式场景，性能成本可接受。
- [窗口矩形部分出屏导致裁切区域缩小] -> 返回实际可见区域截图，并在日志或错误处理中明确该情况。

## Migration Plan

- 第一步：替换窗口截图底层实现为屏幕快照。
- 第二步：接入窗口矩形裁切逻辑和边界收敛处理。
- 第三步：验证识别设置页面仍能正常预览、保存和匹配模板。
- 第四步：运行打包构建，确认应用整体行为未回归。
- 回滚方式：恢复原始目标窗口直接截图实现，不影响其余识别设置能力。

## Open Questions

- 屏幕快照是否需要覆盖虚拟桌面多屏区域，还是先以当前主显示器可见区域为基线。
- 是否要在截图结果中显式标记“窗口当前部分不可见”，供上层页面提示用户。
