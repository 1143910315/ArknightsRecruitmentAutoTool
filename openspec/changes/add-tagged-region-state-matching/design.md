## Context

当前识别设置中的每个区域只保存一张参考图片和一个自由文本标签，数据结构和匹配结果都默认“一区域只有一种命中状态”。这无法覆盖公开招募界面中同一区域会出现多种合法标签图片的场景，也无法把命中结果直接映射为统一的公开招募标签。此次改动需要同时调整前端区域编辑体验、后端模板存储格式、图片存储布局和匹配结果结构，并且要保证整套标签定义在程序内是共享的，而不是散落在多个页面中各自维护。

## Goals / Non-Goals

**Goals:**
- 让单个识别区域可以保存多个状态截图，每个状态截图都绑定一个公开招募标签。
- 提供整个程序共享的公开招募标签分类与枚举，避免多处重复维护。
- 在匹配时按区域内多个状态逐个比对，并返回命中的标签状态信息。
- 保持现有识别设置主流程可用，即仍然是先选窗口、截图、圈区域、再保存模板。

**Non-Goals:**
- 不在本次设计中实现 OCR 或自动识别公开招募标签文字。
- 不扩展新的图片相似度算法，仍沿用当前精确图片比对逻辑。
- 不改造公开招募推荐算法本身，只提供它未来可复用的统一标签集合。
- 不处理旧模板的自动迁移工具界面，只要求运行时兼容或最小迁移。

## Decisions

### 1. 将“区域参考图”升级为“区域状态集合”
每个区域不再只持有单个参考图路径，而是持有多个状态项；每个状态项包含唯一 ID、公开招募标签、参考图路径和创建时间等元数据。

选择原因：
- 区域坐标仍然是一份，共享同一裁切范围；只有截图状态和标签是多份，模型更贴近实际需求。
- 比为每个状态复制一份完整区域定义更节省存储，也更容易在 UI 上组织。

备选方案：
- 把每个状态拆成独立区域。未采用，因为这会重复保存相同坐标，且不利于用户理解“同一区域的不同状态”。

### 2. 将公开招募标签抽为共享目录数据，而不是页面内写死
在前后端共享的应用层中维护标准标签分组：职业、部署位置、特性标签、干员资历，并要求识别设置页面从该目录读取可选项。

选择原因：
- 用户明确要求这些标签是整个程序共同的。
- 后续公开招募页面、识别结果页和推荐流程都可以复用同一来源，避免枚举漂移。

备选方案：
- 继续在识别设置页面内单独维护一份标签数组。未采用，因为会制造第二份标签真值源。

### 3. 匹配结果返回“命中的状态标签集合”，并保留未命中状态
区域匹配时遍历该区域下的所有状态图，对每个状态执行现有图片比对；结果中返回命中的状态 ID 与标签，并保留区域级结果，供页面展示“哪个标签匹配成功”。

选择原因：
- 用户需要看到具体命中的标签，而不是只有 true/false。
- 保留区域级结构可以减少前端展示改动，前端只需把区域结果展开为状态命中列表。

备选方案：
- 只返回第一个命中的标签。未采用，因为同一区域理论上可能有多个状态图完全相同或同样命中，丢弃信息会让结果不透明。

### 4. 旧模板按单状态区域兼容读取
若读取到旧模板中的单一 `referencePath`/`label` 结构，则在加载时将其映射为一个默认状态项，避免旧模板立即失效；新保存的模板统一写入多状态结构。

选择原因：
- 现有识别设置功能已经落地，直接破坏旧模板会导致回归。
- 运行时兼容比一次性强制迁移更低风险。

备选方案：
- 只支持新结构并要求手工重建模板。未采用，因为用户已有模板数据时成本过高。

## Risks / Trade-offs

- [模板结构同时支持旧版和新版] -> 通过读取时归一化为统一内存结构，写入时只输出新结构，降低分叉成本。
- [同一区域状态数增多会让匹配次数上升] -> 当前区域数量和状态数都在人工配置范围内，可接受；后续如有需要再加短路或缓存。
- [多个状态可能使用同一标签] -> 先允许，避免强行限制用户流程；界面可以提示但不阻断。
- [共享标签目录被多个模块依赖] -> 将标签目录保持为纯静态数据和简单查询接口，避免引入额外耦合逻辑。

## Migration Plan

- 第一步：定义共享公开招募标签目录和新的区域状态数据结构。
- 第二步：更新模板读写逻辑，使旧模板可读、新模板按多状态格式保存。
- 第三步：更新识别设置页面，支持在区域下新增多个状态截图并选择标签。
- 第四步：更新匹配结果结构和展示，显示每个区域命中的标签状态。
- 第五步：执行模板加载、保存、匹配和页面交互验证。
- 回滚方式：恢复单状态区域结构和旧匹配结果格式，同时保留未迁移的旧模板文件不变。

## Open Questions

- 对于同一区域内重复标签的多个状态，界面是否需要显式警告，还是仅允许保存。
- 匹配展示是显示全部命中标签，还是默认仅突出一个“首选命中”标签；本次先按显示全部命中设计。
