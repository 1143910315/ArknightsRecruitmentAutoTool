## Context

当前项目已经具备三块可复用能力：公开招募页面可以基于已选标签实时计算组合结果；识别设置可以保存窗口模板、区域相对坐标和多状态标签图；后端可以对指定窗口执行区域裁切，并返回每个区域命中的状态标签列表。缺口在于这些能力尚未串成一个持续运行的公开招募识别流程，尤其没有把模板匹配结果转换为“当前招募标签”的确定性输出，也没有处理多个同标题同类名窗口时该识别哪一个实例。

这次改动同时涉及 Go 后端窗口定位与匹配聚合、Wails 前后端桥接、以及公开招募页面状态生命周期，因此需要先固定识别契约。主要约束包括：
- 自动识别必须基于已有识别模板工作，不能绕开模板配置直接写死区域。
- 同一轮识别只有在所有配置区域都恰好命中一个状态图时才允许产出标签。
- 当多个窗口标题和类名相同，不能只按标题或类名做模糊命中，否则会把标签识别到错误窗口。
- 公开招募页面已经存在手动选标签和组合计算逻辑，新能力应复用这条结果链，而不是再做一份并行逻辑。
- 轮询需要串行执行，且每轮结束后固定等待 500ms，避免识别调用重叠。

## Goals / Non-Goals

**Goals:**
- 为公开招募页面新增模板驱动的自动识别开关和识别状态循环。
- 定义一个严格的识别聚合结果：全区域覆盖、单区域唯一命中、成功后输出标签集合，失败则保持 no-op。
- 在多个同标题同类名窗口并存时，稳定解析出与模板对应的窗口实例；若无法唯一解析，则拒绝识别。
- 让自动识别成功结果直接驱动公开招募页面的标签选择，并复用既有组合计算与展示逻辑。
- 保证整个轮询链路无重入，且识别完成后统一等待 500ms 再进入下一轮。

**Non-Goals:**
- 不自动点击游戏窗口中的公开招募标签，自动勾选仅作用于应用内公开招募页面。
- 不引入 OCR、模糊相似度调参或新的图像匹配算法，本次仍沿用现有精确图片匹配基础。
- 不扩展模板编辑器本身的核心交互，识别设置页面只需继续产出可用模板。
- 不实现复杂任务编排、后台常驻识别服务或跨页面的自动化流程配置中心。

## Decisions

### 1. 新增面向公开招募的聚合识别接口，而不是让页面直接消费原始区域匹配结果
后端在现有 `MatchRecognitionTemplate` 基础上新增一层公开招募识别接口，输入为模板 ID，输出为窗口解析结果、区域唯一匹配状态、识别到的标签列表以及失败原因枚举。页面不再自己解释“某区域命中多个状态图算什么”，而是只消费已经归一化的识别结果。

选择原因：
- 公开招募页面需要的是“本轮标签是否可用”，而不是识别设置页使用的调试级区域明细。
- 把严格匹配规则集中到后端，可以避免前端复制窗口匹配、区域聚合和 no-op 判定逻辑。
- 后续若识别结果还要被其他页面复用，这个接口可以作为稳定边界。

备选方案：
- 页面直接调用 `MatchRecognitionTemplate` 再自行判断。未采用，因为这会让前端承担过多模板匹配细节，并增加重复逻辑。

### 2. 模板增加实例定位元数据，并按“精确句柄优先、实例特征回退、歧义即失败”解析目标窗口
保存模板时除了标题、类名和截图尺寸外，再记录可用于识别实例的元数据，例如捕获时的窗口句柄、进程 ID、窗口矩形和截图尺寸。自动识别时先尝试复用仍然有效的原始句柄；若句柄失效，再从当前顶层窗口中筛选同标题同类名候选，并结合进程 ID、窗口尺寸/位置等特征回退匹配；若仍有多个等价候选，则本轮返回歧义失败，不进入图像匹配。

选择原因：
- 用户已经明确提出需要处理多个同标题同类名窗口，单靠标题和类名不足以定位正确实例。
- 句柄在同一运行期内最稳定，失效时再回退到实例特征，可以降低误识别风险。
- “歧义即失败”比猜测某个窗口更安全，符合本次 no-op 优先的原则。

备选方案：
- 仅使用标题和类名查找第一个匹配窗口。未采用，因为在多开场景下不可预测。
- 要求用户每次在公开招募页手动重新选窗口。未采用，因为会破坏自动识别连续性。

### 3. 公开招募识别采用“全区域唯一命中”契约，并把区域标签聚合为页面选中标签
对模板中的每个区域执行现有图片匹配后，只有在每个区域都恰好命中一个状态图时才判定本轮识别成功；任一区域 0 命中或多命中，都视为整轮失败。成功后按区域命中的状态标签聚合出公开招募标签集合，并去重后写回页面选中标签。

选择原因：
- 用户明确要求“全部区域均发生匹配，且仅匹配一个图片，否则不做任何事”。
- 唯一命中可以保证识别结果稳定，不会把同一区域多个候选标签混入公开招募组合计算。
- 识别成功后直接生成标签集合，能与现有公开招募结果逻辑自然衔接。

备选方案：
- 允许部分区域缺失时返回部分标签。未采用，因为会让结果失真并与用户要求冲突。
- 同一区域多命中时默认取第一个。未采用，因为这种隐式优先级不可解释且容易误选。

### 4. 公开招募页面的自动识别循环由前端页面状态驱动，并保证串行轮询
自动识别开关、轮询定时器和最近一次识别状态都归公开招募页面持有。开关开启后，页面发起一轮识别；只有当前一轮 Promise 完成后，才在 500ms 后安排下一轮。页面卸载、切换页面或关闭开关时立即取消后续调度。

选择原因：
- 轮询只在公开招募页面有意义，把生命周期绑定到页面最简单，也能避免后台常驻任务泄漏。
- 串行调度比固定间隔 `setInterval` 更安全，可以确保截图和识别不会并发重叠。
- 页面天然需要展示识别开启/关闭和最近结果状态，状态集中更利于维护。

备选方案：
- 在后端启动常驻 goroutine 轮询。未采用，因为停止条件、前端可见状态和资源释放都会更复杂。
- 前端使用固定频率 `setInterval`。未采用，因为识别耗时不可控，容易出现重入。

### 5. 页面沿用单一 `selectedTags` 真值源，自动识别只是新增一种写入方式
公开招募页面继续使用现有 `selectedTags` 作为唯一标签选择状态。手动点击标签和自动识别都通过同一状态更新入口写入，组合枚举、结果分组和展示逻辑保持不变。自动识别失败时不清空也不改写当前标签，只有成功识别时才以本轮结果替换选中标签。

选择原因：
- 现有组合计算已经围绕 `selectedTags` 工作，复用它可以把改动限制在输入端。
- “失败不改写”与 no-op 语义一致，也避免识别抖动导致页面不断清空重建。
- 单一真值源可以减少“手动标签”和“自动标签”两套状态漂移。

备选方案：
- 维护单独的 `recognizedTags` 再在视图层合并。未采用，因为状态来源会分叉，页面行为更难推理。

## Risks / Trade-offs

- [模板缺少足够实例指纹，导致多窗口下仍可能歧义] -> 保存模板时补充句柄、进程 ID 和窗口矩形，并在歧义时明确返回失败而不是猜测。
- [现有精确图片匹配对缩放或轻微遮挡敏感] -> 本次保持严格 no-op，先保证不会误识别；后续如需放宽，再单独扩展算法。
- [轮询频率过高可能带来持续截图开销] -> 使用串行执行和 500ms 完成后延迟，避免堆积调用；必要时在页面中展示忙碌状态。
- [自动识别与用户手动选标签可能发生竞争] -> 自动模式开启时通过统一写入口覆盖选中标签，关闭后再恢复纯手动操作。
- [识别模板不唯一或不存在时用户不知道为何不工作] -> 返回明确失败原因并在页面显示模板缺失、窗口歧义或匹配失败状态文案。

## Migration Plan

- 第一步：扩展模板元数据与窗口枚举/解析逻辑，提供公开招募自动识别后端接口。
- 第二步：在后端实现严格区域唯一匹配聚合，并返回标签列表与失败原因。
- 第三步：在公开招募页面新增自动识别开关、轮询状态和识别结果提示。
- 第四步：把识别成功结果写入现有 `selectedTags`，复用已有组合计算与渲染。
- 第五步：验证多窗口、无模板、区域多命中、区域漏命中、开关关闭和 500ms 串行轮询等场景。
- 回滚方式：移除公开招募页面自动识别入口和后端聚合接口，保留现有手动公开招募页面与识别设置能力不变。

## Open Questions

- 当存在多个已保存识别模板时，公开招募页面默认使用哪一个模板：最后一次在识别设置中加载的模板，还是后续需要新增显式模板选择入口。
- 自动模式开启时，是否需要把手动标签按钮显式禁用，还是允许点击但会被下一轮识别结果覆盖。
- 页面是否需要展示最近一次识别失败的细分原因与时间，还是只显示简化状态提示。
