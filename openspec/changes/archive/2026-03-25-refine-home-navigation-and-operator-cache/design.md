## Context

当前应用已经具备两类核心界面：主页应用壳和干员数据页。主页导航目前是固定左侧栏，视觉风格以暖色为主；干员数据页能够从 wiki 页面抓取并展示结果，但数据仍停留在运行时内存中，图片也没有下载和缓存。因此，本次变更同时涉及前端导航交互、视觉主题、后端本地文件存储以及数据加载策略。

这次设计需要满足以下约束：
- 导航交互应继续保持桌面端多页面切换能力，但要释放更多横向空间，因此更适合抽屉式导航而不是固定侧栏。
- 干员数据和图片需要持久化到本地，便于下次启动或再次打开页面时优先展示缓存结果。
- 本地缓存不能改变 wiki 原始条目的顺序，因为后续比对、人工核查和与线上页面同步时依赖该顺序一致性。
- 本次设计只要求“优先本地展示”，不要求自动后台刷新，也不要求复杂的缓存失效策略。

## Goals / Non-Goals

**Goals:**
- 将主页导航改为 Element Plus Drawer，并保持页面切换逻辑清晰稳定。
- 将整体页面色调调整为浅蓝-白风格，并在主页和干员数据页上保持一致。
- 在抓取 wiki 数据后，将干员结构化数据和图片一起保存到本地。
- 在打开干员数据页面时优先加载本地缓存，并保持展示顺序与 wiki 页面解析顺序一致。

**Non-Goals:**
- 本次设计不引入自动定时刷新或缓存过期淘汰机制。
- 不增加新的数据筛选、搜索、分页或排序功能。
- 不要求把本地缓存同步到云端或跨设备共享。
- 不改变 wiki 数据源本身，也不保证适配未来所有页面结构变化。

## Decisions

### 1. 使用“顶部触发器 + Drawer 菜单”替代固定左侧栏
应用壳保留统一主视图，但导航入口改为顶部或页面内的菜单触发器，点击后打开 Element Plus Drawer，Drawer 内包含页面列表与当前激活状态。

选择原因：
- 相比固定侧栏，Drawer 能释放更多主内容宽度，尤其适合数据表格页面。
- Element Plus Drawer 能与当前组件体系保持一致，减少额外状态管理成本。

备选方案：
- 继续保留固定侧栏，仅压缩宽度。未采用，因为这只缓解空间问题，不能改善移动端和窄窗口下的适配体验。

### 2. 本地缓存采用“JSON 数据文件 + 图片目录”结构
后端在用户数据目录或应用缓存目录下维护一个专用目录，结构化干员数据写入单个 JSON 文件，图片按干员标识或顺序保存为本地文件，并在数据记录中引用本地路径。

选择原因：
- 这种结构简单、可调试、易于后续迁移，不需要立即引入数据库。
- 图片与数据分离存放，便于局部更新和排查单张图片下载失败的问题。

备选方案：
- 将图片直接嵌入 JSON 为 base64。未采用，因为文件体积会明显膨胀，不利于本地维护和后续增量更新。

### 3. 缓存读取优先于远程抓取，远程抓取仅在用户显式触发时执行
干员数据页初始化时先读取本地缓存；如果缓存存在则立即展示；如果缓存不存在则显示空状态。远程抓取仍由用户点击按钮触发，并在成功后覆盖本地缓存。

选择原因：
- 这满足“打开页面时优先展示本地数据”的体验目标。
- 显式抓取能避免每次打开页面都联网，也减少对第三方站点的无意请求。

备选方案：
- 页面打开时自动先展示缓存再后台刷新。未采用，因为这会引入额外的同步状态和冲突处理，超出本次范围。

### 4. 顺序字段按解析顺序显式保存
解析 `.contentDetail` 节点时，为每条干员记录保存顺序索引，并在写入缓存和页面渲染时按该索引保持原始顺序，即使未来引入其他展示字段也不默认重新排序。

选择原因：
- 显式顺序字段比依赖“数组自然顺序”更稳健，写入和读取逻辑更容易验证。
- 能避免未来新增排序、筛选或数据转换时无意改变原始顺序。

备选方案：
- 仅依赖 JSON 数组写入顺序。未采用，因为中间处理一旦插入映射或去重逻辑，顺序更容易丢失。

### 5. 图片下载失败不阻塞数据缓存写入
单张图片下载失败时，结构化干员数据仍然应写入本地缓存，记录中对应的本地图片路径可为空或保留远程源地址，前端需要能够处理缺图展示。

选择原因：
- 数据内容比图片更关键，不能因为个别图片失败导致整个缓存流程失败。
- 这能提高第一次抓取的整体成功率，并方便后续补充重试机制。

备选方案：
- 任意图片失败就整体回滚缓存。未采用，因为这会让缓存系统对外部站点波动过于敏感。

## Risks / Trade-offs

- [本地缓存目录选择不当导致权限问题] -> 使用 Wails/Go 可写的用户数据目录或应用缓存目录，并在写入失败时给出明确错误。
- [图片数量较多导致首次抓取变慢] -> 允许图片下载独立失败，不阻塞数据缓存落盘，并在 UI 上显示加载反馈。
- [Drawer 交互增加一次点击成本] -> 用清晰的菜单触发器和当前页面标识降低切换成本。
- [未来主题变更再次扩大样式修改范围] -> 将浅蓝-白主题集中到全局变量或主要容器样式中，减少分散覆盖。
- [缓存和线上数据不一致] -> 明确页面默认优先本地，线上刷新仅在用户主动触发时覆盖缓存。

## Migration Plan

- 第一步：重构前端应用壳，将固定左侧导航改为 Drawer，并同步调整浅蓝-白主题样式。
- 第二步：扩展后端干员数据模型，增加顺序索引、本地图片路径和缓存文件结构。
- 第三步：实现缓存读取、JSON 写入和图片下载逻辑，并在抓取成功后落盘。
- 第四步：在干员数据页接入“优先本地加载”的初始化流程和缓存展示状态。
- 第五步：验证 `go test ./...`、`pnpm build` 和 Wails 打包流程仍然可用。
- 回滚方式：如果缓存逻辑不稳定，可临时停用本地优先加载并保留远程抓取路径；如果 Drawer 体验不符合预期，可回退到固定导航但不影响缓存模块存在。

## Open Questions

- 本地缓存目录应使用用户数据目录还是项目工作目录，是否需要在 UI 中暴露缓存位置。
- 图片文件名是否直接使用干员名，还是使用顺序索引加干员名的组合以减少命名冲突。
- 是否需要在后续版本中加入“清空缓存”或“重新下载全部图片”的操作。
