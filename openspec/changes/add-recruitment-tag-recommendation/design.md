## Context

当前仓库是一个基于 Wails 的 Windows 桌面应用，后端使用 Go，前端使用 Vue。现有 `main.go` 仍主要暴露窗口枚举、鼠标位置和高亮窗口等通用能力，尚未形成围绕《明日方舟》公开招募的业务流程，因此需要用一次跨前后端的设计将输入、识别、规则计算和结果展示串联起来。

本次变更需要满足几个约束：
- 产品必须以离线桌面工具为主，不依赖在线服务才能完成基础分析。
- 软件只提供识别与推荐，不自动点击游戏或注入进程。
- OCR 结果存在误识别风险，流程必须允许用户看到识别结果并进行修正。
- 公开招募推荐依赖标签、干员与规则数据，数据结构需要能够独立维护和后续更新。

## Goals / Non-Goals

**Goals:**
- 建立从截图输入到推荐输出的完整业务链路。
- 明确前后端职责边界，使后续编码能分阶段推进。
- 为 OCR 识别失败、低置信度和规则解释提供可用的降级路径。
- 将招募标签和干员规则从界面逻辑中分离，便于维护本地数据集。

**Non-Goals:**
- 本次设计不定义具体 OCR 库的最终选型代码接入细节。
- 不包含自动控制游戏窗口、自动选择标签或其他自动化操作。
- 不覆盖公开招募之外的任何《明日方舟》玩法。

## Decisions

### 1. 采用“四段式流水线”架构
应用按照 `截图输入 -> 图片预处理/识别 -> 规则计算 -> 结果展示` 的顺序执行，每一步输出明确结构化数据，避免把 OCR 文本、业务规则和界面展示耦合在一起。

选择原因：
- 便于单独替换 OCR 或图片预处理实现。
- 便于在识别失败时直接复用截图和中间结果进行人工修正。
- 便于后续给规则引擎或数据集增加测试。

备选方案：
- 直接在前端串联所有逻辑。未采用，因为图片处理、OCR 封装和规则数据更适合在 Go 侧管理，且会让前端承担过多平台细节。

### 2. 截图输入优先支持“文件导入”，保留“屏幕采集”扩展位
第一阶段将文件导入定义为必备输入方式，屏幕指定区域采集作为可扩展能力放在相同输入接口之下。

选择原因：
- 文件导入实现最稳定，便于快速验证识别和推荐主链路。
- Windows 下屏幕采集通常涉及更多权限、DPI 和窗口坐标问题，适合作为后续增强。

备选方案：
- 首版强制实时截屏。未采用，因为这会把 MVP 依赖绑定到平台截图实现，拖慢整体交付。

### 3. 识别结果必须带置信度和人工修正入口
OCR 输出不仅包含标签文本，还要包含匹配到的标准标签、置信度和异常状态。当前端检测到未识别、重复或低置信度标签时，必须允许用户手动修正后再继续推荐。

选择原因：
- 公开招募的价值在于正确性，错误标签比无结果更危险。
- 用户修正一次的成本远低于做出错误招募决策。

备选方案：
- 只显示 OCR 原文，不允许修正。未采用，因为这会把最终正确性责任完全推给用户，同时无法保证规则计算输入合法。

### 4. 推荐引擎基于本地标准化数据集运行
标签、干员、星级、公开招募可出干员关系等数据应以本地静态数据文件或内置资源形式管理，规则引擎基于标准标签 ID 进行组合分析，而不是直接依赖 OCR 原文。

选择原因：
- 本地数据便于离线运行和版本控制。
- 使用标准标签 ID 能降低 OCR 同义词、空格或错字对规则计算的影响。

备选方案：
- 直接用识别文本做字符串匹配。未采用，因为容错性和可维护性都较差。

### 5. 推荐结果按“确定性价值优先”排序
结果排序优先级应以可稳定锁定的最高稀有度、稀有标签价值和候选集合收敛程度为主，而不是简单按候选干员数量排序。结果中还要说明推荐理由，例如“锁定高级资深干员”“可保证至少五星”“包含支援机械”。

选择原因：
- 用户真正关心的是收益和确定性，而不是组合数量。
- 解释性文案有助于建立用户信任，并支持人工复核。

备选方案：
- 仅返回所有组合列表。未采用，因为缺乏优先级和解释，不能真正起到助手作用。

## Risks / Trade-offs

- [OCR 对截图质量敏感] -> 通过要求用户截取公开招募区域、增加预处理步骤并提供人工修正入口降低风险。
- [游戏版本更新导致标签或干员数据过期] -> 将标签和干员关系维护为独立数据文件，后续可单独更新。
- [Windows 屏幕采集实现复杂] -> 将文件导入设为必备路径，屏幕采集作为兼容增强而非首版阻塞项。
- [推荐逻辑若不可解释会降低用户信任] -> 结果输出中强制包含推荐理由、候选干员和风险说明。
- [现有代码与目标产品方向差距较大] -> 在实现阶段先新增独立业务模块，不与现有窗口工具逻辑深度耦合，必要时逐步替换旧接口。

## Migration Plan

- 第一步：保留现有 Wails 工程结构，在前后端新增公开招募分析流程，不直接删除旧窗口工具逻辑。
- 第二步：完成截图导入、OCR 识别和规则推荐的最小闭环后，再决定是否移除与业务无关的窗口工具界面。
- 第三步：将标签和干员数据固定为可维护的数据文件，并建立更新方式。
- 回滚方式：如果新流程不稳定，可临时隐藏入口或回退到只保留截图展示与手动输入标签的版本，不影响基础桌面框架启动。

## Open Questions

- 首版 OCR 采用何种本地方案最合适，是否需要引入额外语言模型或词表增强。
- 干员数据源如何维护，是否直接内置仓库静态文件，还是引入单独生成脚本。
- 屏幕采集是否只支持整屏截图后裁切，还是要支持指定窗口区域。
- 推荐排序中是否需要支持用户自定义偏好，例如“优先新手实用”或“优先高星”。
