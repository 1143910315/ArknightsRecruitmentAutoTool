## Context

当前项目已经具备基于鼠标指向获取窗口、读取窗口标题与类名以及窗口高亮等能力，但这些能力仍停留在“定位窗口”层面，还没有进入“为指定窗口建立识别模板”的配置流。用户需要一个完整的页面，能够选择目标窗口、查看窗口截图、在截图上划分多个识别区域，并为每个区域保存相对位置、标签和参考图片，以便后续识别时判断窗口局部图像是否匹配。

这次改动跨越前端交互、窗口截图、模板数据结构和持久化，因此必须先明确数据模型和页面流程。主要约束包括：
- 区域位置必须以相对窗口截图的坐标保存，避免窗口移动时模板失效。
- 一个窗口模板需要支持多个区域，每个区域都有独立标签与参考图片。
- 用户选中的参考图片不能只停留在前端内存，需要以可复用的形式持久化。
- 后续识别流程要基于这些已保存模板做局部图像匹配，因此当前设计必须把匹配所需字段定义完整。
- 本次以识别设置和模板保存为主，不扩展成完整自动识别任务编排系统。

## Goals / Non-Goals

**Goals:**
- 新增识别设置页面，并接入现有导航。
- 允许用户通过鼠标指向方式选中目标窗口，并展示标题、类名和窗口截图。
- 允许用户在截图上定义多个区域，并为每个区域设置标签和参考图片。
- 持久化窗口模板和区域模板数据，保存相对位置、标签以及参考图片信息。
- 为后续“指定区域图片是否匹配模板图片”提供清晰、可复用的数据结构。

**Non-Goals:**
- 不在本次设计中实现完整自动化识别任务执行流程。
- 不引入复杂的图像识别算法选择、阈值调优面板或批量模板管理系统。
- 不覆盖所有可能的窗口标识策略，只聚焦当前页面保存模板所需的最小稳定标识。
- 不处理云同步、多人共享或跨设备模板迁移。

## Decisions

### 1. 识别设置页面复用现有窗口选择能力，但增加独立截图与模板编辑流程
页面通过已有鼠标指向窗口能力选择目标窗口，再在识别设置页内部完成截图、区域绘制和保存，而不是把这些操作堆叠到现有窗口工具页。

选择原因：
- 识别设置是独立任务流，涉及窗口信息、截图、绘制、模板保存，页面状态明显比窗口工具更复杂。
- 复用底层窗口定位能力，同时避免污染现有窗口工具用途。

备选方案：
- 直接在窗口工具页扩展识别设置区域。未采用，因为会让当前工具页承担过多状态和交互步骤。

### 2. 模板数据以“窗口模板 + 多区域模板”结构持久化
每个窗口模板保存窗口标题、类名、截图尺寸、窗口截图引用和多个区域配置；每个区域配置保存标签、相对矩形坐标和参考图片引用。

选择原因：
- 后续识别按窗口模板进行，区域是窗口模板的天然子对象。
- 坐标和图片引用一起保存，能直接支持局部裁切与匹配。

备选方案：
- 区域模板独立平铺存储。未采用，因为会弱化区域与所属窗口截图之间的关系，后续恢复编辑上下文更困难。

### 3. 区域位置使用归一化相对坐标，而不是绝对像素坐标
区域保存为相对窗口截图宽高的百分比或 0..1 范围坐标，并在界面编辑时再映射回像素。

选择原因：
- 即使窗口截图在不同缩放或重取样场景下尺寸变化，归一化坐标也更稳定。
- 后续匹配时可以按当前截图尺寸重新还原区域框。

备选方案：
- 直接保存绝对像素坐标。未采用，因为窗口截图尺寸一旦变化，模板就更容易失效。

### 4. 参考图片以区域裁切结果为主保存，并允许标签与图片绑定
当用户在截图上确定一个区域后，系统优先使用该区域从窗口截图中裁切出的图片作为参考图保存，而不是要求用户单独上传第二张图片。

选择原因：
- 需求中的“区域所选图片”与截图区域天然一致，直接裁切最符合使用路径。
- 这样可以保证参考图与区域坐标来源一致，减少用户出错。

备选方案：
- 用户手动上传图片作为模板。未采用，因为当前需求强调基于窗口截图进行区域标注，不需要额外上传流程。

### 5. 匹配能力本次定义接口和结果模型，不强行承诺复杂算法
当前设计先定义“给定窗口模板和区域模板，对当前窗口截图对应区域执行图片匹配并返回是否命中及基础结果”的能力边界。算法实现可先以直接像素比较、相似度阈值或简单模板匹配为基础。

选择原因：
- 需求的关键是建立稳定配置流和匹配入口，而不是在 proposal 阶段锁死复杂图像算法。
- 先固定输入输出模型，可以降低后续识别逻辑返工。

备选方案：
- 现在就指定完整算法和阈值体系。未采用，因为当前信息不足，过早绑定实现细节风险高。

## Risks / Trade-offs

- [窗口截图能力可能依赖平台实现细节] -> 在实现时复用并封装现有 Windows 能力，优先验证截图链路是否稳定。
- [区域编辑交互复杂，容易出现误操作] -> 采用截图预览 + 明确的区域列表 + 可重选和删除机制降低编辑负担。
- [模板图片体积增长] -> 只保存区域裁切图而不是整张重复截图，并控制模板目录结构。
- [窗口标题可能变化导致模板不稳定] -> 同时保存类名、标题和其他可用窗口标识，后续匹配时组合使用。
- [归一化坐标换算误差影响匹配] -> 保存截图原始尺寸并在恢复时统一换算规则，避免多套坐标口径。

## Migration Plan

- 第一步：新增识别设置页面和导航入口。
- 第二步：补充后端窗口截图与模板持久化接口。
- 第三步：实现前端窗口选择、截图预览和多区域标注交互。
- 第四步：保存窗口模板、区域模板和区域参考图片。
- 第五步：补充区域匹配入口和基础验证流程。
- 回滚方式：移除识别设置页面和模板相关接口，不影响现有窗口工具和干员数据能力。

## Open Questions

- 窗口截图具体复用现有 Win32 链路还是新增专门截图实现，取决于当前代码中是否已有可复用截图能力。
- 匹配结果是否需要暴露相似度分数，还是只先返回布尔值和基础状态。
- 模板保存目录是否需要与未来识别任务配置共享统一结构。
