## Context

当前应用已经实现了本地优先的干员数据缓存，但缓存目录位于用户缓存目录下，而不是程序运行目录。同时，前端目前通过本地文件 URL 访问缓存图片，这与 Wails 的安全边界和前端运行环境并不稳健。用户现在明确要求缓存跟随程序运行目录，并通过 Wails 的 `AssetsHandler` 或等效资源处理机制向前端暴露本地图片资源。

这个变更跨越缓存目录解析、后端资源路径生成、Wails 应用资源处理配置以及前端图片展示路径，因此需要先固定设计约束。约束包括：
- 缓存目录必须位于程序运行目录下的可预测位置，而不是依赖用户目录。
- 前端不能直接读取本地文件系统，因此本地图片必须通过 Wails 管理的资源入口暴露。
- 已有的本地优先加载、显式远程刷新和源顺序保持行为不能因为目录迁移而退化。
- 运行目录写权限在不同启动方式下可能存在差异，因此需要定义错误反馈与降级策略。

## Goals / Non-Goals

**Goals:**
- 将干员数据 JSON 和图片缓存迁移到程序运行目录下的专用缓存位置。
- 通过 Wails 资源处理能力暴露本地缓存图片给前端展示。
- 保持缓存优先加载、手动刷新和源顺序一致的现有行为。
- 明确目录结构、资源 URL 生成方式和运行目录写入失败时的处理方式。

**Non-Goals:**
- 本次设计不引入新的缓存失效策略或后台自动刷新。
- 不新增图片压缩、缩略图生成或资源版本管理机制。
- 不改变远程 wiki 抓取与解析规则本身。
- 不要求兼容任意只读安装目录场景之外的复杂权限提升流程。

## Decisions

### 1. 缓存目录固定为程序运行目录下的子目录
后端通过可执行文件所在目录或当前程序运行目录计算缓存根目录，并在其中创建专用子目录存放 `operators.json` 和图片资源。

选择原因：
- 这直接满足“缓存保存在程序运行目录”的需求。
- 运行目录内的缓存结构对用户和调试过程都更直观。

备选方案：
- 继续使用用户目录，仅在 UI 中显示路径。未采用，因为这不满足需求本身。

### 2. 本地图片通过 Wails `AssetsHandler` 暴露为可访问路径
Wails 应用层新增本地缓存资源的处理逻辑，把运行目录中的缓存图片映射到前端可访问的应用内路径。后端返回该应用内路径，而不是 `file://` 本地文件 URL。

选择原因：
- 这与 Wails 前端无法直接访问本地文件的运行模型一致。
- 统一由 Wails 托管资源路径，更安全也更可预测。

备选方案：
- 继续返回绝对文件路径。未采用，因为前端无法可靠直接消费本地文件路径。

### 3. 数据记录保留“本地相对路径 + Wails 访问路径”两个层次
缓存写入时记录相对图片路径，读取缓存并返回给前端时再派生出通过 Wails 访问的图片 URL。这样缓存文件本身不依赖机器绝对路径，但前端仍能得到可用资源地址。

选择原因：
- 相对路径更利于缓存目录迁移和打包后的可移植性。
- 把可访问 URL 作为运行时派生值，能降低缓存格式与部署路径耦合。

备选方案：
- 在 JSON 中直接存绝对路径或完整资源 URL。未采用，因为运行目录可能变化，绝对路径不稳定。

### 4. 目录迁移以“新位置优先，不自动复杂迁移”为主
实现阶段优先读取新运行目录缓存；如果新位置不存在则按空缓存处理，用户可通过显式刷新重新建立缓存。除非实现成本很低，否则不强制做用户目录到运行目录的自动迁移。

选择原因：
- 这能保持变更简单，避免跨目录复制带来的权限和一致性问题。
- 用户只需手动刷新一次即可重建缓存，不影响主流程。

备选方案：
- 首次启动自动从旧目录迁移。暂不采用，因为这需要额外处理旧路径发现、复制失败和回滚逻辑。

## Risks / Trade-offs

- [程序运行目录在某些安装位置不可写] -> 写入失败时返回明确错误，并允许用户通过手动刷新感知问题而不是静默失败。
- [Wails 资源映射实现错误导致本地图片无法展示] -> 通过构建后启动验证缓存图片路径可用性。
- [目录变更使旧缓存暂时不可见] -> 接受一次性重新抓取重建缓存，避免复杂自动迁移。
- [资源 URL 与缓存文件结构耦合过深] -> 只在运行时生成 Wails 访问路径，缓存文件内部保留相对路径。

## Migration Plan

- 第一步：调整后端缓存根目录解析逻辑，从用户目录切换到程序运行目录。
- 第二步：调整缓存文件结构和图片路径字段，改为保存相对路径并在读取时派生 Wails 资源 URL。
- 第三步：在 Wails 应用配置中接入本地缓存资源处理入口。
- 第四步：更新前端图片展示逻辑，仅消费 Wails 资源路径。
- 第五步：验证 `go test ./...`、`pnpm build`、`wails build` 和缓存存在时的桌面启动展示行为。
- 回滚方式：如果 Wails 资源处理逻辑不稳定，可临时回退为仅显示无图占位，同时保留运行目录缓存数据。

## Open Questions

- 程序运行目录应以可执行文件目录为准，还是开发运行时的当前工作目录为准。
- 是否需要在后续版本提供从旧用户目录缓存导入到运行目录的显式操作。
- `AssetsHandler` 的路径前缀是否应独立命名为专用缓存资源路由，以避免与内置静态资源混淆。
