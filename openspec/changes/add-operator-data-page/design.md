## Context

当前应用的前端仍是一个以窗口树和 Win32 调试能力为中心的单页面视图，后端 `main.go` 暴露的也是窗口枚举、鼠标定位和高亮等通用接口。为了把产品逐步收敛到《明日方舟》公开招募助手，需要先在界面结构上支持业务页面扩展，并补齐干员基础数据的获取与展示路径。

这次变更同时跨越前端布局、Wails 接口、网络抓取和 HTML 解析四个层面，适合先通过设计文档固定关键决策。约束包括：
- 现有项目是 Wails + Go + Vue 桌面应用，页面切换应尽量复用现有单窗口应用壳，不引入复杂路由框架作为前置条件。
- 干员数据源来自公开网页，页面结构可能变动，解析逻辑必须尽量围绕稳定的 DOM 结构和字段来源设计。
- 初始页面没有数据，数据获取必须由用户显式触发，避免应用启动时无提示联网。
- 这次能力主要为后续公开招募规则计算准备数据，因此需要保留比当前页面展示更多的结构化字段，而不只是名字列表。

## Goals / Non-Goals

**Goals:**
- 将现有单页界面改造成带左侧导航的应用壳，至少支持切换到 `干员数据` 页面。
- 建立从前端按钮触发到 Go 后端抓取 wiki 页面并解析干员数据的最小闭环。
- 定义统一的干员数据模型，既能支撑当前页面展示，也能复用于后续公开招募推荐逻辑。
- 明确加载中、空状态、抓取失败和抓取成功后的页面行为。

**Non-Goals:**
- 本次设计不要求实现持久化缓存或本地数据库，数据可先保留在当前运行时内存中。
- 不覆盖公开招募推荐算法本身，只准备干员数据获取与浏览能力。
- 不保证适配 wiki 页面未来所有结构调整，只要求基于当前稳定结构可解析全部干员条目。
- 不移除现有窗口工具能力，只要求它们在新的导航结构下被保留或隔离。

## Decisions

### 1. 前端采用“应用壳 + 视图切换”而非立即引入完整路由
前端保留单个 `App.vue` 入口，新增左侧导航和基于当前选中页面的视图切换，先把现有窗口工具页和新 `干员数据` 页组织到统一应用壳中。

选择原因：
- 当前前端规模很小，引入完整 Vue Router 会增加样板和状态同步成本，但收益有限。
- 视图切换足以支撑当前的两页结构，并为后续真正需要多页面时再升级保留空间。

备选方案：
- 立即引入 Vue Router。未采用，因为当前桌面应用没有深链接、浏览器历史或复杂嵌套路由需求。

### 2. 数据抓取放在 Go 后端执行，由前端只负责触发和展示
`获取干员数据` 按钮调用新的 Wails 绑定方法，由 Go 侧执行 HTTP GET 和 HTML 解析，再把标准化结果返回前端。

选择原因：
- Go 侧更适合管理网络请求、解析依赖和后续可能的本地缓存逻辑。
- 可以避免在前端直接处理跨域、HTML 文本体积和解析库选择问题。
- 后续公开招募规则引擎本来就更可能落在 Go 侧，数据模型放在后端更连贯。

备选方案：
- 由前端直接 fetch 页面并解析 DOM。未采用，因为 Wails 前端侧需要处理网页请求和解析依赖，且不利于后续复用到规则层。

### 3. 解析逻辑围绕 `.contentDetail`、`.picText`、`.tags .tagText` 和 `data-param*` 字段构建
解析器以每个 `.contentDetail` 元素为一条干员记录，优先提取名称、星级、展示标签，并把 `data-param1` 拆分为元数据数组后再映射到结构化字段。

选择原因：
- 这些字段直接对应用户给出的样例，并且在页面中承担明确的语义角色。
- 同时保留“原始 metadata 列表”和“映射后的常用字段”可以在页面结构微调时降低回归成本。

备选方案：
- 只读取可见文本标签，不解析 `data-param1`。未采用，因为这会丢失职业、性别、阵营、公开招募标记等后续规则需要的重要信息。

### 4. 干员数据模型区分“展示字段”和“招募元数据”
后端返回的干员记录至少包含 `name`、`rarity`、`displayTags`、`metadata`、`isPublicRecruitable` 等字段。页面展示只依赖前几项，但完整元数据一并返回，避免后续为推荐逻辑重复抓取和重构模型。

选择原因：
- 当前页面只需要展示基础信息，但业务价值在于把数据沉淀成规则输入。
- 结构化返回可以减少未来扩展时的破坏性改动。

备选方案：
- 只返回当前 UI 所需字段。未采用，因为后续公开招募能力几乎一定会再次改接口。

### 5. 运行时状态采用“空状态、加载中、成功、失败”四态模型
`干员数据` 页面用明确的状态机管理当前视图，初始为空；点击抓取后进入加载中；成功后展示结果；失败后显示错误且不覆盖已有成功数据。

选择原因：
- 这能避免“空数据”和“加载失败”被混淆。
- 如果未来支持重新抓取，保留旧数据并显示错误更符合桌面工具的稳定预期。

备选方案：
- 失败时直接清空列表并回到初始页。未采用，因为这会让用户失去已成功获取的数据上下文。

## Risks / Trade-offs

- [wiki 页面 DOM 结构变化导致解析失效] -> 解析器尽量依赖稳定 class 和 data 属性，并为缺失字段返回明确错误。
- [首次联网抓取较慢，用户误以为无响应] -> 前端必须展示加载状态，并在完成后给出成功或失败反馈。
- [字段映射假设不完整，影响后续招募逻辑复用] -> 同时保留原始 metadata 列表和映射后的常用字段，避免一次性丢失信息。
- [现有窗口工具 UI 与新业务页混杂，导致界面复杂] -> 通过左侧导航隔离页面，而不是继续在同一视图中追加区块。
- [后端新增网络依赖增加测试难度] -> 将抓取和解析拆成独立函数，后续可用固定 HTML 样本做解析测试。

## Migration Plan

- 第一步：在前端建立统一应用壳和左侧导航，把当前窗口工具视图收纳为一个页面。
- 第二步：新增 Go 侧干员数据模型、抓取方法和解析逻辑，并通过 Wails 暴露给前端。
- 第三步：在 `干员数据` 页面接入按钮、状态展示和数据列表渲染。
- 第四步：补充基于样例 HTML 的解析验证，并确认现有应用仍能正常构建。
- 回滚方式：如果新页面或抓取逻辑不稳定，可以临时隐藏 `干员数据` 导航入口，不影响原有窗口工具功能启动。

## Open Questions

- 是否需要在后续版本中把抓取到的干员数据持久化到本地文件，避免每次启动都重新获取。
- `data-param1` 中部分位置相关字段是否需要进一步标准化成枚举，还是先保留原始字符串即可。
- 成功获取后是否需要补充搜索、筛选或分页，还是首版直接全量列表展示。
