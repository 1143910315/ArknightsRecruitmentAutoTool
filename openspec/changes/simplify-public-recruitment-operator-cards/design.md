## Context

当前公开招募页面会在 Combination 结果中为每位干员展示名称、星级文本、职业/阵营和标签列表，信息密度偏高。页面已经具备本地干员数据加载能力，并且同一仓库中的干员数据页已经证明前端可以通过 Wails 接口把本地缓存图片解析为 `data:<mime>;base64,...` 形式进行展示。

这次变更只涉及 `frontend/src/components/PublicRecruitmentPage.vue` 的结果呈现层，不改动标签组合生成、干员匹配、自动识别、缓存读取和后端数据结构。约束是必须继续兼容无图场景，并且边框颜色要严格匹配产品给定的星级映射。

## Goals / Non-Goals

**Goals:**
- 在 Combination 结果中将干员卡片收敛为“图片 + 名称”的最小展示。
- 复用现有缓存图片读取机制，为公开招募页面提供可渲染的 base64 图片源。
- 通过稳定的星级到颜色映射，为干员图片边框提供一致的视觉编码。
- 保持现有组合筛选、排序和命中逻辑不变。

**Non-Goals:**
- 不修改后端接口或缓存文件格式。
- 不调整公开招募页面的标签选择、自动识别或组合排序规则。
- 不改变干员数据页的展示方式。
- 不引入新的 UI 组件库或额外依赖。

## Decisions

### 1. 在公开招募页面本地解析并缓存图片源
复用 `OperatorDataPage.vue` 已验证过的图片加载方式：根据 `localImagePath` 调用 `GetCachedOperatorImage`，再在前端拼装为 base64 data URL。这样不需要新增后端接口，也能保证 Combination 结果直接显示本地缓存图片。

备选方案：
- 直接复用干员数据页中的图片状态存储。未采用，因为当前页面之间没有共享图片缓存状态，强行抽共享会扩大这次变更范围。
- 在结果卡片中继续不展示图片。未采用，因为需求明确要求显示 base64 图片。

### 2. 用星级映射函数统一生成图片边框色
在公开招募页面新增一个纯函数，根据 `operator.rarity` 返回固定颜色：1-2 星 `#dedede`，3 星 `#618bf5`，4 星 `#8960ce`，5 星 `#f0a94d`，6 星 `#f0e028`。样式层通过 CSS 变量或内联样式消费该结果，避免在模板里分散条件判断。

备选方案：
- 通过多个 rarity class 写死模板分支。未采用，因为会增加模板噪音，后续调整颜色时也不集中。

### 3. 卡片内容只保留图片区域和名称
删除卡片中的星级文本、职业/阵营文案和标签 chip。卡片布局改为以图片为主的紧凑网格，名称位于图片下方。无图时保留占位区域和名称，避免因缺图导致布局塌陷或结果缺失。

备选方案：
- 保留星级文本辅助说明。未采用，因为边框颜色已经承担稀有度表达，文本会重新增加视觉噪音。

## Risks / Trade-offs

- [Risk] 公开招募页面新增图片读取会带来一次额外的异步加载过程 → Mitigation: 只按唯一 `localImagePath` 去重加载，并缓存到页面级状态，避免同一轮结果重复请求。
- [Risk] 少数干员缺少本地图片时卡片可能显得不完整 → Mitigation: 保留固定尺寸的无图占位和名称展示，保证结果仍可识别。
- [Risk] 仅靠边框颜色表达星级，对色弱用户的信息冗余较少 → Mitigation: 当前需求明确要求仅展示图片和名字，本次按需求收敛；若后续需要无障碍增强，可再补充可选文本或 tooltip。

## Migration Plan

这是纯前端展示改动，无数据迁移。

实施步骤：
1. 在公开招募页面增加图片加载与缓存状态。
2. 调整结果卡片模板，只保留图片和名称。
3. 增加 rarity-to-border-color 映射与对应样式。
4. 运行前端构建验证页面能正常编译。

回滚方式：恢复 `PublicRecruitmentPage.vue` 中原有卡片模板与样式即可。
