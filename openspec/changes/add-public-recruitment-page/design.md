## Context

当前项目已经具备首页导航、干员数据页、本地缓存优先加载以及公开招募干员数据基础字段，但还没有专门服务公开招募筛选的页面。现有数据里已经包含职业、部署位置、招募标签和资深标签等字段，因此这次改动的核心不是新增数据源，而是在前端把这些字段组织成公开招募所需的标签选择与组合结果展示能力。

本次需求存在几个明确约束：
- 可选标签集合是固定的，分为职业、部署位置、特性标签、干员资历四类。
- 用户最多同时选择 5 个标签。
- 选中多个标签时，需要生成全部非空子组合，而不是只展示完整组合命中的结果。
- 没有匹配干员的组合不展示。
- 页面应基于本地已加载的干员数据工作，不改变当前干员数据抓取、缓存和顺序处理逻辑。

## Goals / Non-Goals

**Goals:**
- 新增一个公开招募页面，提供固定标签集合的可视化选择能力。
- 在前端限制最多选择 5 个标签，并给出清晰反馈。
- 基于已选标签生成全部非空组合，并稳定展示每个有效组合对应的干员列表。
- 让组合匹配逻辑复用当前已加载的干员数据，不引入新的后端接口或远程请求。
- 保持结果展示顺序稳定，便于用户快速比较完整组合和子组合的命中差异。

**Non-Goals:**
- 不改变干员抓取、缓存位置、图片加载或数据持久化方式。
- 不引入新的招募算法、概率估算或最优推荐排序功能。
- 不支持自定义标签集合、跨分类联动配置或超过 5 个标签的高级模式。
- 不在本次设计中新增后端筛选 API，组合计算全部在前端完成。

## Decisions

### 1. 公开招募页面作为独立导航页面接入现有前端结构
新增独立页面组件，并挂入当前导航体系，而不是把筛选逻辑混入干员数据页。

选择原因：
- 公开招募是独立任务流，页面职责清晰。
- 可以复用已有本地缓存数据加载能力，同时避免让干员数据页承担过多交互状态。

备选方案：
- 在干员数据页内部增加筛选区域。未采用，因为会把“数据浏览”和“招募筛选”两种用途混在一起，页面复杂度会明显上升。

### 2. 标签集合使用前端静态配置，匹配基于干员记录字段归一化执行
职业、部署位置、特性标签、资历标签使用固定前端配置渲染；匹配时把每个干员可参与公开招募筛选的标签整理为集合，再判断是否覆盖某个组合。

选择原因：
- 当前需求里的标签列表是明确且固定的，静态配置成本最低。
- 组合匹配只需要集合包含关系，不需要引入复杂索引结构。

备选方案：
- 运行时从全部干员数据动态提取标签。未采用，因为会引入不受控标签，且不能保证和产品定义的可选标签一致。

### 3. 组合生成采用前端幂集枚举，并过滤空组合和空结果组合
当用户选择 N 个标签时，页面生成所有 1..N 长度的非空组合；再按组合长度降序、原始选择顺序稳定排序，并过滤没有命中干员的组合。

选择原因：
- N 最大为 5，最多 31 个组合，前端直接枚举足够简单且性能可控。
- 长组合优先展示更符合公开招募使用习惯，同时保留子组合结果。

备选方案：
- 只展示完整组合命中结果。未采用，因为需求明确要求显示所有有效子组合。
- 后端生成组合结果。未采用，因为当前数据已在前端，额外往返没有收益。

### 4. 结果分组保持组合维度清晰，并在组内保持干员原始顺序
每个有效组合渲染为一个结果分组，显示组合标签标题和对应干员列表；组内干员顺序沿用当前数据顺序，不额外按星级或名称重排。

选择原因：
- 页面目标是帮助用户理解“哪个标签组合能招到哪些人”，组合应是主视角。
- 保持原始顺序可以与已有数据展示保持一致，减少额外排序规则带来的歧义。

备选方案：
- 按干员聚合展示其命中组合。未采用，因为用户实际操作是先看组合，再看该组合能出谁。

### 5. 超过 5 个标签时阻止新增选择，不自动替换已有选择
当已选数量达到 5，再点击未选标签时不生效，并通过 UI 提示说明限制；已选标签仍可取消。

选择原因：
- 这与“最多五个”约束直接一致，行为最可预测。
- 避免自动挤掉旧标签造成用户困惑。

备选方案：
- 自动移除最早选择的标签。未采用，因为会破坏组合语义且不可预期。

## Risks / Trade-offs

- [标签匹配字段口径不一致] -> 在实现时统一从职业、部署位置、展示标签和资深标签归一化出可匹配标签集合，并只针对公开招募可用干员做筛选。
- [组合数量随选择增加而上升] -> 通过 5 个标签上限将组合数控制在 31 个以内，前端枚举可接受。
- [结果分组过多影响可读性] -> 使用“长组合优先、空结果过滤”的规则降低视觉噪音。
- [页面在无缓存数据时不可用] -> 复用现有本地缓存状态提示，在无数据时引导用户先抓取干员数据。

## Migration Plan

- 第一步：新增公开招募页面组件与导航入口。
- 第二步：整理固定标签配置和标签选择状态管理。
- 第三步：实现基于本地干员数据的标签归一化、组合生成和结果过滤逻辑。
- 第四步：完善空状态、五标签限制提示和结果展示样式。
- 第五步：验证页面在有缓存、无缓存、单标签、多标签和五标签上限场景下的行为。
- 回滚方式：移除公开招募页面入口与组件，不影响现有干员数据抓取和浏览能力。

## Open Questions

- 组合标题是否需要额外展示命中人数徽标，还是仅显示标签文本即可。
- 公开招募页面是否直接复用现有干员卡片样式，还是使用更紧凑的列表样式。
- 是否需要对“高级资深干员”这类资历标签做醒目视觉强调。
