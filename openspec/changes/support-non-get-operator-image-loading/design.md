## Context

当前干员数据缓存已经迁移到程序运行目录，本地图片路径也已经改为相对路径和应用内逻辑 URL。但是上一轮实现仍然默认前端通过 GET 方式请求 Wails 处理后的图片路径，而你当前的实际约束是该处理入口不支持 GET，只支持 POST/PUT。这样一来，缓存图片虽然存在，但前端渲染链路仍然不成立。

这次设计需要解决“如何在不依赖 GET 资产访问的前提下，把本地缓存图片交给前端渲染”的问题。约束包括：
- 前端不能直接读取本地文件系统，也不能继续依赖 GET 资产路径。
- 缓存目录仍然需要保留在程序运行目录下，不回退到用户目录。
- 干员数据页面仍要保持本地优先加载、显式刷新和源顺序一致。
- 新的图片获取方式应尽量复用 Wails 已支持的前后端通信机制，而不是增加独立服务进程。

## Goals / Non-Goals

**Goals:**
- 为缓存图片提供一个不依赖 GET 的、Wails 兼容的读取接口。
- 让前端基于该接口把缓存图片转为可渲染形式并显示在干员列表中。
- 保持运行目录缓存、列表顺序和本地优先加载行为不变。
- 明确缺图、读取失败和大图片数据传输时的处理方式。

**Non-Goals:**
- 本次设计不改变远程抓取、缓存目录位置或缓存写入流程。
- 不新增图片编辑、压缩或缩略图缓存机制。
- 不引入独立 HTTP 服务或外部文件服务器。
- 不解决所有资源类型的非 GET 访问问题，仅覆盖干员缓存图片。

## Decisions

### 1. 使用 Wails 绑定方法直接返回图片内容而不是依赖资产 URL
后端新增专门的图片读取方法，前端通过 Wails 绑定直接请求指定缓存图片，并获得可渲染的内容结果，例如 base64 数据、字节数组或带 MIME 的数据结构。

选择原因：
- Wails 绑定调用本身不依赖浏览器直接发起 GET 请求。
- 这条链路与现有前后端通信模型一致，落地成本低于构建新的资源服务层。

备选方案：
- 继续尝试用 POST/PUT 驱动资产处理器返回图片。未采用，因为图片标签和标准浏览器渲染链天然依赖 GET，不适合作为主方案。

### 2. 前端在数据加载后按需解析图片响应为可展示 URL
前端拿到后端图片响应后，将其转换为浏览器可用的展示形式，例如 `data:` URL 或 Blob URL，并把该结果绑定到图片组件上。

选择原因：
- 这样可以避开静态资源 GET 依赖，同时保留标准 `<img>` 渲染方式。
- 图片是否存在、是否解码成功都可以在前端显式控制并回退。

备选方案：
- 后端直接返回完整 HTML 或内嵌样式。未采用，因为这会让前端展示层和图片传输层耦合过深。

### 3. 图片加载采用“数据先到，图片后补”的渐进式策略
干员基础数据仍先整体从缓存加载，图片显示改为在前端渲染列表后再逐项请求或批量请求缓存图片内容。无法取到图片时不影响基础数据展示。

选择原因：
- 列表主体与图片渲染解耦，失败时影响范围更小。
- 这与已有缺图兜底逻辑兼容，不需要重新设计整个数据页状态机。

备选方案：
- 后端一次性在缓存数据中内联所有图片内容。未采用，因为这会显著放大缓存数据返回体积。

### 4. 返回结果中保留图片元信息，支持缺图和 MIME 判断
后端图片读取接口返回结构化结果，至少包含是否命中、图片 MIME 类型和可传输的图片内容。前端根据这些元信息生成对应展示 URL 或直接进入缺图回退。

选择原因：
- MIME 信息对前端正确生成 `data:` URL 或 Blob 很关键。
- “命中/未命中”分离后，错误处理可以更清晰。

备选方案：
- 仅返回裸字符串内容。未采用，因为前端无法可靠区分缺图、文本错误和真实图片类型。

## Risks / Trade-offs

- [图片通过绑定返回会增加 IPC 传输开销] -> 采用按需加载或渐进式加载，避免首次渲染时把所有图片一起塞进主数据响应。
- [base64 或数据 URL 体积较大] -> 控制图片尺寸来源为现有缓存图，不额外放大原图，并保留后续换成 Blob URL 的空间。
- [图片请求失败导致界面闪烁] -> 前端保留稳定的缺图占位，不让失败影响列表主体渲染。
- [接口设计不清导致前端状态复杂] -> 使用结构化图片响应，明确区分成功、缺失和错误。

## Migration Plan

- 第一步：新增后端缓存图片读取接口，基于运行目录缓存路径返回图片内容与 MIME 信息。
- 第二步：在前端数据页中移除对 GET 资产路径的依赖，改为通过绑定调用解析本地图片。
- 第三步：保留现有缺图占位逻辑，并在图片加载失败时静默降级。
- 第四步：验证 `go test ./...`、`pnpm build`、`wails build` 和缓存存在时的桌面端图片展示行为。
- 回滚方式：如果新图片读取流程不稳定，可暂时只显示无图占位，保留缓存数据和列表展示能力。

## Open Questions

- 图片内容采用 base64 字符串还是字节数组返回，对当前 Wails 绑定层最稳妥。
- 前端是否需要做简单的图片结果缓存，避免重复请求同一张图片。
- 是否应一次暴露单张图片读取接口和批量图片读取接口，还是先做单张版本。
